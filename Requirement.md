项目总览：CT071-3-3-DDAC 云应用开发项目完整任务书

1. 项目背景
- 课程：CT071-3-3-DDAC - DESIGNING & DEVELOPING CLOUD APPLICATIONS
- 形式：小组项目（3-4人）
- 总分：50分
- 行业：物业管理行业（已选定）

2. 核心学习目标
- CLO2：设计符合性能标准且适合云部署的应用程序。
- CLO3：选择适当的云服务以构建高可用性和成本效益的应用程序。

3. 任务一 (Task 1) - 基础开发与云部署 (30分)
- 提交周：第10周
- 目标：开发并部署一个基础的、可工作的物业管理Web应用到AWS云。

3.1. 技术栈要求
- 前端：HTML5, CSS3, JavaScript (必须实现响应式设计)
- 后端：ASP.NET Core (C#) (必须使用)
- 云数据库：AWS RDS (MySQL/PostgreSQL) 或 AWS DynamoDB (必须使用)
- 计算服务：Amazon Elastic Beanstalk (推荐) 或 Amazon EC2 (必须将应用部署至此)
- 禁止：使用本地数据库

3.2. 核心功能列表 (最小可行产品)
- 用户认证与授权：业主、物业管理员、维修工三个角色的注册、登录、权限控制。
- 业主门户：
  - 提交报修单（含问题描述、图片上传功能）
  - 查看个人报修单列表及状态（待处理、已分配、维修中、已完成）
  - 查看个人资料和房产信息
- 物业管理员后台：
  - 系统仪表盘（显示关键指标，如待处理工单数）
  - 全功能工单管理（查看所有工单、分配工单给维修工）
  - 用户管理（增删改查业主、维修工账户）
  - 房产信息管理
- 维修工门户：
  - 查看分配给自己的工单
  - 更新工单状态（如开始维修、完成维修）

3.3. 数据库实体 (需创建)
- Users表：存储用户信息（ID、用户名、密码哈希、角色、邮箱等）
- Properties表：存储房产信息（ID、地址、单元号、业主ID等）
- RepairTickets表：存储报修工单（ID、标题、描述、状态、创建时间、创建人ID、分配人ID、图片URL等）
- (可选) PaymentRecords表：存储缴费记录

3.4. 任务一提交物清单
- 1. PDF报告：由PowerPoint转换，包含封面、工作量矩阵、数据库创建截图、部署步骤截图、用户界面截图、应用URL和测试账号密码。
- 2. 源代码ZIP包：完整的项目源代码（前端、后端、数据库脚本等）。
- 3. 口头演示：在第8-10周期间向讲师进行10分钟系统演示。

4. 任务二 (Task 2) - 云服务集成与监控 (20分)
- 前提：必须完成任务一
- 提交周：第14周
- 目标：在任务一应用基础上，集成无服务器架构和其他云服务，并实现性能监控。

4.1. 各部分要求
- Part 1: 云架构设计 (5分)
  - 要求：设计新的云架构图，将任务一的单体架构演进为包含微服务和无服务器概念的混合架构。需在报告中与旧架构图对比并说明变更理由。

- Part 2: 无服务器架构与云服务集成 (10分)
  - 强制集成：必须使用 Amazon API Gateway 和 AWS Lambda。
  - 可选集成（至少选一种）：必须集成以下至少一项服务：
    - Amazon S3 (对象存储，用于存储报修图片等)
    - Amazon SNS (消息通知服务，用于工单状态变更的短信/邮件通知)
    - Amazon SQS (消息队列服务，用于管理工单分配等异步任务)
  - 示例：将“提交报修”功能改造为：前端 -> API Gateway -> Lambda函数 -> 写数据库 -> 触发SNS通知业主和维修工。

- Part 3: 性能监控 (5分)
  - 要求：使用AWS监控工具（如CloudWatch, X-Ray）监控服务器、Lambda函数及应用的性能。
  - 交付：在报告中包含监控仪表板的截图，并对性能数据（如CPU/内存利用率、函数执行时间、X-Ray服务地图）进行分析讨论。

4.2. 任务二提交物清单
- 1. Word文档报告：最大60页，需包含：
  - 新旧架构图对比及变更说明。
  - 云服务集成（API Gateway, Lambda, S3/SNS/SQS）的代码和配置截图与说明。
  - 用户手册（更新后的功能截图与说明）。
  - 性能监控截图与分析。
  - 每个团队成员的个人反思。
  - 工作量分配矩阵。
